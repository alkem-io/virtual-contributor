# Research: Pipeline Reliability and BoK Resilience

**Feature Branch**: `story/020-pipeline-reliability`
**Date**: 2026-04-15

## Research Tasks

### R1: Why not retry on asyncio.TimeoutError (zombie thread analysis)

**Context**: The LLM adapter uses `asyncio.wait_for(asyncio.to_thread(sync_call), timeout=120)`. When the timeout fires, `asyncio.wait_for` cancels the coroutine returned by `asyncio.to_thread`, but Python threads cannot be cancelled or interrupted. The thread continues running until the sync LLM call completes or the process exits.

**Findings**:

Each timed-out call creates a zombie thread: a thread that occupies a slot in the thread pool executor, holds its network connection, and continues consuming resources, but whose result will never be read. When the adapter retried on timeout, it submitted a *new* thread for the same call.

With `MAX_RETRIES=3` and `DocumentSummaryStep` running 8 concurrent summarizations, a single pipeline where all LLM calls time out creates:

- Round 1: 8 zombie threads (timed out) + 8 retry threads = 16 total
- Round 2: 8 more zombie threads + 8 retry threads = 32 total
- Round 3: 8 more zombie threads = 40 total

The default Python thread pool is `min(32, os.cpu_count() + 4)`, typically 8-16 workers. Even with the explicit 32-worker pool, 40 zombies from a single pipeline exceed capacity. With two concurrent pipelines, the pool is exhausted and all `asyncio.to_thread` calls (including ChromaDB operations) block indefinitely -- a deadlock.

**Decision**: Do not retry on `asyncio.TimeoutError`. Immediately raise `TimeoutError` to preserve thread pool capacity for other tasks.
**Rationale**: The zombie thread is already consuming a pool slot. Retrying adds more zombies. The only safe response is to fail fast and let the pipeline handle the per-document failure gracefully (via `_SummaryResult.error`).
**Alternatives considered**: (a) Kill the thread -- impossible in CPython without `ctypes` hacks that corrupt interpreter state. (b) Use a separate thread pool for LLM calls -- adds complexity without addressing the root cause (zombie accumulation). (c) Reduce retry count for timeouts -- delays the deadlock but does not prevent it.

### R2: Why try/finally over try/except for background embedding tasks

**Context**: `DocumentSummaryStep.execute()` creates background `asyncio.Task` objects for embedding and then processes more results. If an exception occurs between task creation and the await loop, the tasks are orphaned.

**Findings**:

The original code structure was:

```python
for result in results:
    # ... create tasks ...
    embed_tasks.append(task)

# Await tasks
for task in embed_tasks:
    await task
```

If processing a result raised an exception (e.g., unexpected `None` in a success result), the loop would exit early. Previously this used `assert` statements which would raise `AssertionError`. Tasks created before the failing result would never be awaited. The tasks would eventually complete but:

1. Their thread pool slots are occupied until completion (resource leak).
2. Any exceptions in the tasks are silently swallowed (debugging difficulty).
3. If many tasks accumulate, thread pool pressure increases across pipelines.

`try/finally` guarantees the `asyncio.gather(*embed_tasks, return_exceptions=True)` runs regardless of whether the result-processing loop completes normally or raises. `return_exceptions=True` prevents individual task failures from masking the original exception.

The `assert` statements were also replaced with graceful error handling (appending to `context.errors`), but the `try/finally` is the structural guarantee that tasks are always awaited even if new failure modes are introduced later.

**Decision**: Wrap task creation + result processing in `try`, await tasks in `finally`.
**Rationale**: Defensive resource management. The `finally` block is a structural guarantee independent of which exceptions might occur. `try/except` would require catching specific exceptions and re-raising after cleanup, which is more fragile.
**Alternatives considered**: (a) `try/except Exception` with re-raise -- works but is more verbose and easy to break if new exceptions are added. (b) `contextlib.ExitStack` -- overkill for this pattern. (c) Structured concurrency via `asyncio.TaskGroup` -- requires Python 3.11+ semantics that cancel sibling tasks on failure, which is not the desired behavior here (we want all tasks to complete).

### R3: Why inline persist over a separate step for BoK

**Context**: The BoK summary is generated by `BodyOfKnowledgeSummaryStep`, then embedded by `EmbedStep`, then stored by `StoreStep`. If `EmbedStep` or `StoreStep` fails, the BoK (which may have taken 10-40 minutes to generate) is discarded.

**Findings**:

The pipeline sequence is: BoKStep -> EmbedStep -> StoreStep -> OrphanCleanupStep. A failure in EmbedStep or StoreStep (network error, ChromaDB restart, etc.) causes the BoK to be lost. The next re-ingestion may skip BoK generation if change detection finds no changes, leaving the store permanently without a BoK entry until content actually changes.

Inline persistence (embed + store inside BoKStep) eliminates this window:

1. BoK is generated (LLM call).
2. BoK is immediately embedded and stored using the same ports.
3. The BoK chunk is still appended to context, so the deferred EmbedStep/StoreStep path acts as a no-op (chunk already has embedding, StoreStep upserts).

If inline persist fails, the BoK chunk is still appended to context with `embedding=None`, so the deferred path handles it. This makes inline persist a best-effort optimization with a graceful fallback.

**Decision**: Persist BoK inline when both `embeddings_port` and `knowledge_store_port` are provided.
**Rationale**: The BoK is the most expensive LLM artifact in the pipeline. Persisting it immediately after generation protects the investment against downstream failures. The fallback to the deferred path maintains backward compatibility.
**Alternatives considered**: (a) Separate `PersistBoKStep` -- adds a new step type to the pipeline, increasing cognitive load. The ports would need to be threaded through the engine, which is not how the current step protocol works. (b) Retry the entire pipeline on StoreStep failure -- the pipeline is not idempotent due to early ACK, and retry would re-run all LLM calls. (c) Checkpoint the BoK to disk -- adds a disk I/O dependency and cleanup complexity.

### R4: Why character-based over count-based section grouping

**Context**: BoK generation uses refine-pattern summarization where each section becomes one LLM call. With 20 documents, that is 20 sequential calls. At ~2 min/call (gemma-26b), this takes ~40 minutes.

**Findings**:

Count-based grouping (e.g., "merge every 4 sections into 1") treats all documents equally regardless of size. A 500-char page and a 50,000-char article would be grouped together, potentially creating groups that exceed the model's context window or are wildly uneven.

Character-based grouping (`max_section_chars=30000`) produces groups of roughly equal input size:

- Small pages (500-2000 chars): 15-60 per group -> 1 round instead of 15-60
- Medium articles (5000-10000 chars): 3-6 per group -> 1 round instead of 3-6
- Large documents (30000+ chars): 1 per group -> unchanged

The 30000 default was chosen because:
- Most summarization models have 32K-128K context windows.
- The refinement prompt includes the running summary (~10K chars) + system prompt (~500 chars) + the section content.
- 30K leaves 2K-98K headroom depending on the model.

For a typical 20-page website with 2000 chars/page = 40K total, this reduces from 20 rounds to 2 rounds: an 90% reduction in LLM calls.

**Decision**: Group sections by cumulative character count with a configurable `max_section_chars` (default 30000, minimum 1000).
**Rationale**: Adapts to heterogeneous document sizes. Keeps each group within model context limits. Reduces wall-clock time proportionally to the grouping factor.
**Alternatives considered**: (a) Count-based grouping -- uneven group sizes, context window overflow risk. (b) Token-based grouping -- requires a tokenizer, which is model-specific and adds a dependency. Character count is a reasonable proxy (1 token ~= 4 chars). (c) Parallel refinement -- violates the sequential nature of refine-pattern (each round depends on the previous summary).

### R5: Partial summary tradeoffs

**Context**: When `_refine_summarize` fails on round N after successfully completing rounds 1..N-1, the partial summary is returned instead of raising an exception.

**Findings**:

**Benefits**:
- Preserves completed LLM work. With slow models, rounds 1..N-1 may represent 10-30 minutes of computation.
- A partial summary covering 80% of the corpus is strictly better than no summary at all for RAG retrieval quality.
- The document summaries that contribute to the BoK are already individually useful; the BoK adds a cross-document synthesis layer. A partial synthesis is more useful than none.

**Risks**:
- The partial summary may be biased toward early documents (the ones processed in rounds 1..N-1).
- The summary length budget is progressive (40% to 100%), so the partial summary may be shorter than the final budget.
- If the failure is caused by content in later documents (e.g., encoding issues), the summary silently omits those documents.

**Mitigations**:
- A warning log is emitted with the round number and count of completed rounds, so operators can identify which documents were excluded.
- The partial summary is deterministic given the same input order, so re-ingestion after fixing the root cause produces the complete summary.
- If round 1 fails (no partial summary exists), the exception propagates normally -- there is no risk of returning an empty string.

**Decision**: Return partial summary when at least one round completed. Propagate exception on first-round failure.
**Rationale**: Partial information is strictly better than no information for the BoK use case. The progressive budget means even a half-done summary has reasonable length.

## Summary of Decisions

| Topic | Decision | Key Rationale |
|-------|----------|---------------|
| Timeout retry | Do not retry on `asyncio.TimeoutError` | Each retry creates another zombie thread; pool exhaustion causes deadlock |
| Background task safety | `try/finally` with `asyncio.gather(..., return_exceptions=True)` | Structural guarantee tasks are always awaited regardless of exception path |
| BoK persistence | Inline embed + store when ports available | Protects expensive LLM work against downstream step failures |
| Section grouping | Character-based with `max_section_chars=30000` | Adapts to document size, stays within context window, reduces rounds 60-90% |
| Partial summary | Return partial on mid-stream failure, raise on first-round failure | Partial is strictly better than nothing; first-round failure means no useful data |
| Thread pool sizing | Explicit 32-worker `ThreadPoolExecutor` | Matches worst-case workload: 8 concurrent summarizations x 3 pipelines + ChromaDB ops |
| Embeddings truthiness | `is not None and len(...) > 0` | Empty list `[]` is truthy but semantically means "no embeddings" |
| StoreStep dedup | Keep last occurrence per storage ID | Prevents ChromaDB duplicate ID errors without losing data |
